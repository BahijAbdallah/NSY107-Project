────────────────────────────────────────────────────────────────
1. PROJECT GOAL
────────────────────────────────────────────────────────────────
This project demonstrates how to:
  (a) Build a secure REST API with authentication and request validation
  (b) Deploy it behind AWS API Gateway for throttling and access control
  (c) Collect access logs and use Machine Learning to detect attack traffic
      in real time (anomaly detection)

The pipeline is:
  Client
    → AWS API Gateway  (throttling, logging, routing)
    → EC2 Node.js Backend  (JWT auth, validation, business logic)
    → CloudWatch Logs  (or local simulation)
    → CSV Dataset
    → Preprocessing
    → ML Models (Random Forest, Isolation Forest, Autoencoder)
    → Best Model
    → Alert: "NORMAL" or "ANOMALY DETECTED"


────────────────────────────────────────────────────────────────
2. ARCHITECTURE EXPLAINED
────────────────────────────────────────────────────────────────

2.1 CLIENT
-----------
  Any HTTP client (web browser, mobile app, Python script).
  Sends requests to the API Gateway URL.

2.2 AWS API GATEWAY
--------------------
  A managed AWS service that sits in front of the backend.
  WHY: It handles security and traffic management BEFORE requests
  even reach our server.

  What it does in this project:
    - ROUTING:        forwards /public, /login, /secure, /orders to EC2
    - THROTTLING:     limits each IP to N requests per second (rate limiting)
                      Returns HTTP 429 Too Many Requests if exceeded
    - USAGE PLANS:    can enforce per-API-key quotas
    - CLOUDWATCH LOGS: logs every request (IP, route, status, latency)
                       This is our data source for AI anomaly detection

2.3 EC2 — NODE.JS / EXPRESS BACKEND  (backend/server.js)
----------------------------------------------------------
  Our application server running on an Amazon EC2 virtual machine.
  Built with Node.js and the Express framework.

  Routes:
    GET  /public   — no auth needed, returns public message
    POST /login    — validates username+password, returns JWT token
    GET  /secure   — requires valid JWT in Authorization header
    POST /orders   — requires JWT + validates itemName and quantity

  Additional features:
    - Request logger: logs every request as JSON to stdout
      (stdout → CloudWatch Logs on EC2)
    - JWT middleware: checks every protected route for a valid token
    - Input validation: rejects bad payloads with clear error messages

2.4 JWT (JSON WEB TOKEN)
------------------------
  What is a JWT?
    A JWT is a signed token that proves who you are.
    After login, the server gives you a token like:
      eyJhbGciOiJIUzI1NiJ9.eyJ1c2VybmFtZSI6ImFkbWluIn0.SIGNATURE
    It has 3 parts separated by dots: Header.Payload.Signature
    The server signs it with a secret key.

  Why use it?
    - No need to send password on every request
    - Stateless: server does not need to store sessions
    - Expiration: tokens expire after 1 hour (reduces risk if stolen)
    - Easy to validate: server re-checks the signature each time

  How it works here:
    1. Client: POST /login  { username, password }
    2. Server: if valid → return JWT token
    3. Client: store the token
    4. Client: GET /secure  Authorization: Bearer <token>
    5. Server: verify token signature and expiry → allow or reject

2.5 THROTTLING / RATE LIMITING
--------------------------------
  What is it?
    Limiting how many requests a client can make in a time window.
    Example: max 100 requests per minute per IP.

  Why?
    - Prevents DDoS attacks (flooding the server)
    - Prevents brute-force login attempts
    - Protects backend cost and performance

  In this project:
    - API Gateway handles rate limiting automatically
    - Returns HTTP 429 when limit is exceeded
    - Our attack script triggers this to test the detection

2.6 REQUEST VALIDATION
-----------------------
  POST /orders validates:
    - itemName: must be a non-empty string
    - quantity: must be a positive integer

  If invalid → HTTP 400 with a clear error message.
  This prevents malformed data from reaching the database/business logic.

2.7 CLOUDWATCH LOGS
--------------------
  AWS CloudWatch is a logging and monitoring service.
  Our backend writes JSON log lines to stdout.
  On EC2, these are captured and forwarded to CloudWatch.

  Each log entry contains:
    timestamp, IP, method, route, status code, latency_ms

  These logs are the RAW DATA for our AI pipeline.
  We export them to CSV format for analysis and model training.

3. DATASET PIPELINE
────────────────────────────────────────────────────────────────

3.1 build_dataset.py
---------------------
  Since CloudWatch is not available in the local development environment,
  this script generates a statistically realistic simulated dataset.

  It creates:
    (a) data/api_gateway_dataset.csv  — one row per HTTP request
        Columns: timestamp, ip, route, method, status, latency,
                 response_size, label

    (b) data/api_gateway_features.csv  — aggregated per IP / 5-minute window
        This is what the ML models actually train on.

  What is a "5-minute window"?
    Instead of analyzing each request individually, we group requests
    by IP address and 5-minute time bucket. We then compute statistics
    for that group. This gives us behavioral patterns.

  Raw data characteristics (v3 — 4 attack types, deduplicated):
    Normal traffic (label=0):
      - 50 different IPs spread across a full day (24 h)
      - Mostly GET /public and GET /secure
      - Mostly 200 OK (72%) with realistic occasional 401 (6%) and 429 (2%)
      - Latency: ~80ms average (realistic network speed)
      - Volume: ~1-4 requests per 5-min window per IP

    Attack traffic (label=1) — four distinct types:
      Type A — Flood (16 IPs):
        - 4 high-volume bursts across the day (hours 4, 10, 16, 22)
        - 5-15 requests per 5-min burst (reduced to OVERLAP with high-volume normal)
        - 25% 401, 25% 429, 20% 403 status codes
        - Higher latency (~200ms, server under stress)

      Type B — Slow Brute Force (12 IPs):
        - 2-5 requests per 5-min window — OVERLAPS with normal
        - 88% status 401 — failed login attempts
        - Spread across 18 hours — no time clustering
        - Low request_count makes this STEALTHY

      Type C — Credential Stuffing (6 IPs):
        - 5-20 requests per 5-min window
        - 40% succeed (200 OK) using stolen credentials — VERY STEALTHY
        - 40% fail (401) — mix of success and failure
        - Latency similar to normal (~100ms)

      Type D — Recon Scan (4 IPs):
        - 3-10 requests per 5-min window
        - Visits ALL 4 routes in each session (unique_routes signal)
        - Mixed status codes, normal-looking latency

  Deduplication (v3 fix):
    Original dataset had 4,155 duplicate windows (same IP + time bucket
    computed twice). These were removed. Final dataset: 3,558 unique windows.

3.2 Aggregated Features (v5 — 23 rate-normalized + historical features)
------------------------------------------------------------------------
  Per IP / 5-minute window.

  IMPORTANT DESIGN DECISION (v3):
    Raw count features (request_count, status_401_count, etc.) were REMOVED.
    Why? Raw counts create trivially easy separation: flood has 50+ requests,
    normal has 1-4. The model just thresholds on volume, not behavior.
    That would fail on real traffic where volume varies widely by user.

    Instead, all features are RATE-NORMALIZED (percentages, ratios, per-route):
    They capture behavioral patterns, not absolute volume.

  Feature group 1 — Rate / behavioral (12 features, introduced v3):
  Feature                | What it measures
  -----------------------|-----------------------------------------------
  error_rate             | fraction of requests that failed (4xx/5xx)
  unauthorized_rate      | fraction that got 401 Unauthorized
  throttle_rate          | fraction that got 429 Too Many Requests
  forbidden_rate         | fraction that got 403 Forbidden
  success_rate           | fraction that got 200/201 OK
  requests_per_route     | average requests per unique route accessed
  avg_gap_seconds        | average seconds between consecutive requests
  min_gap_seconds        | smallest gap between requests (speed signal)
  latency_std            | standard deviation of response times
  post_ratio             | fraction of requests using POST method
  login_ratio            | fraction targeting the /login endpoint
  unique_status_codes    | count of distinct status codes seen

  Feature group 2 — Continuous (3 features, original):
  avg_latency            | mean response latency in ms
  max_latency            | peak latency in ms
  unique_routes          | distinct endpoints accessed

  Feature group 3 — Route diversity (1 feature, introduced v4):
  route_entropy          | Shannon entropy of the route distribution.
                         | 0.0 when all requests go to one route (brute force).
                         | log2(4)=2.0 when all 4 routes are hit equally (recon).
                         | Normal users: ~0.7-1.2.

  Feature group 4 — Within-window route sequence (2 features, introduced v5):
  route_switches_ratio   | Consecutive route changes / (n-1).
                         | Recon alternates routes → 1.0.
                         | Brute force stays on /login → 0.0.
  repeated_route_ratio   | Fraction of requests going to an already-visited route.
                         | Recon visits each route once → 0.0.
                         | Brute force repeats /login → 0.75+.

  Feature group 5 — Per-IP rolling / lag features (5 features, introduced v5):
  prev_requests_per_route     | requests_per_route of the PREVIOUS window for this IP.
  delta_requests_per_route    | Change in requests_per_route vs previous window.
  rolling2_unique_routes      | Mean unique_routes over the 2 most recent PAST windows.
  rolling2_route_entropy      | Mean route_entropy over the 2 most recent PAST windows.
  rolling2_error_rate         | Mean error_rate over the 2 most recent PAST windows.

  No-leakage guarantee: all rolling features use shift(1) before rolling(),
  so the current window is NEVER included in its own rolling mean. Only
  past windows are used. This mirrors production deployment where only
  historical data is available at inference time.

  Why rolling features help Recon detection:
    A single Recon window with 2-3 requests can look like normal browsing.
    But its IP's HISTORY shows consistent multi-route access: the rolling
    unique_routes and route_entropy remain elevated over multiple windows,
    a pattern that is absent for normal IPs. This is the key insight of v5.

  label                  | Ground truth: 0=normal, 1=attack (NOT a feature)
  attack_type            | flood/slow_brute/credential/recon (metadata ONLY,
                         | never passed to any model)


────────────────────────────────────────────────────────────────
4. PREPROCESSING  (src/preprocess.py)
────────────────────────────────────────────────────────────────

Steps:
  1. Load api_gateway_features.csv
  2. Drop rows with missing values
  3. TIME-BASED SPLIT at the 80th percentile of window_start:
     - Train: earliest 80% of time windows
     - Test:  latest 20% of time windows
     This tests TEMPORAL GENERALIZATION — can the model detect
     attacks it hasn't seen yet, in future time periods?
     This is far more realistic than random shuffling.
  4. Scale all numeric features with StandardScaler
     (FITTED ON TRAINING SET ONLY — never contaminated by test data)
  5. Create TWO variants of the split:

  Supervised split (for Random Forest):
    - Training set: first 80% by time, all labels
    - Test set:     last 20% by time

  Unsupervised split (for Isolation Forest + Autoencoder):
    - Training set: NORMAL rows ONLY from the 80% train window
    - Test set:     ALL rows in the last 20% (normal + attack)
    - Labels are NOT used during training, only for evaluation

  WHY TIME-BASED SPLIT?
    Random shuffling leaks future information into the past.
    In production, a model is trained on past data and deployed
    on future data. Time-based split tests this honestly.
    It also prevents the same IP's windows from appearing in both
    train and test (which would inflate metrics).

  WHY TWO DIFFERENT SPLITS?
    Supervised learning (Random Forest) needs labeled examples of
    BOTH classes. Unsupervised learning (Isolation Forest, Autoencoder)
    learns only from normal behavior. Anything different = anomaly.
    This is more realistic where labeled attack data may not exist.

  Output also saved:
    data/X_test_raw.csv       — unscaled test features (for error analysis)
    data/attack_type_test.csv — attack type labels for test set (analysis only)
    results/feature_list.txt  — the 15 feature column names used for ML

5. MACHINE LEARNING MODELS  (src/train_models.py)
────────────────────────────────────────────────────────────────

5.1 Random Forest (SUPERVISED)
--------------------------------
  Algorithm: Ensemble of decision trees. Each tree votes.
  Majority vote = final prediction (0 or 1).

  Why supervised?
    Random Forest needs LABELED data (examples of normal AND attack).
    It learns rules like:
      "if unauthorized_rate > 0.8 AND login_ratio > 0.5 → attack"

  Hyperparameter tuning (v3 — TimeSeriesSplit, anti-overfitting):
    Uses RandomizedSearchCV with:
      n_iter=25    — tries 25 random combinations
      cv=TimeSeriesSplit(n_splits=3) — TIME-AWARE cross-validation
      scoring="f1" — optimizes F1, not accuracy

  5.1b Hist GradientBoosting (SUPERVISED — v5 best model)
---------------------------------------------------------
  Algorithm: Boosted ensemble of shallow decision trees. Each tree corrects
  the errors of the previous one (gradient descent on the loss function).

  Why add it?
    GBM is often better than RF at learning FEATURE INTERACTIONS (e.g., the
    combination of high unique_routes AND low login_ratio AND route_entropy=2.0
    is a recon signal that a single tree split cannot capture).
    Adding it gives a direct comparison on the same TimeSeriesSplit protocol.

  Training (v5 — with 23 features including rolling/sequence features):
    - RandomizedSearchCV, 20 iterations, TimeSeriesSplit(3)
    - `class_weight='balanced'`, L2 regularization tuned
    - Best config: max_depth=4, learning_rate=0.05, min_samples_leaf=20, l2=0.1
    - Train F1: 1.0000 | CV F1: 0.9809 | Test F1: 0.9677
    - Gap (CV-test): 0.0132 | Gap (train-test): 0.0323 [ACCEPTABLE]

  NOTE on Train F1=1.0000:
    The training gap vs CV (1.0000 - 0.9809 = 0.0191) is small. This indicates
    the model fits training data well but does not memorize it — the CV and test
    scores are close. The larger dataset (5,312 vs 3,550 in v4) and the rolling
    features being highly discriminative together explain the near-perfect train fit.
    The gap relative to TEST (0.0323) remains well within the 0.10 threshold.

  Result (v5):
    GBM outperforms RF: F1=0.9677 vs RF F1=0.9591; FPR=0.0157 vs RF FPR=0.0182.
    GBM is selected as best model in v5.

  Result (v4 comparison):
    v4: RF best (F1=0.915, Recon recall=71%)
    v5: GBM best (F1=0.9677 before tuning, Recon recall=90.2%) — clear improvement.

  WHY TimeSeriesSplit instead of regular k-fold?
      Regular k-fold shuffles the data, so fold 3 can "see" time window 200
      while training on time window 250 (future leakage). This gives an
      overly optimistic CV score. TimeSeriesSplit trains on windows 0..k and
      validates on windows k+1..end — exactly how real deployment works.
      This gives an HONEST estimate of generalization.

    Search space (v3 — constrained to prevent overfitting):
      n_estimators:      [200, 300, 500]
      max_depth:         [5, 8, 10, 12, 15]   (no None — prevents full memorization)
      min_samples_split: [5, 10, 15]
      min_samples_leaf:  [3, 5, 8, 10]         (larger = less overfitting)
      max_features:      ["sqrt", "log2"]
      class_weight:      ["balanced", "balanced_subsample"]

    Best params found: n_estimators=500, max_depth=8, min_samples_split=10,
                       min_samples_leaf=5, max_features="sqrt",
                       class_weight="balanced"

  Overfitting check (v3):
    Train F1 (on full training set): 0.9399
    CV F1   (TimeSeriesSplit mean):  0.9310
    Test F1 (held-out future data):  0.9103
    Overfitting gap (train - test):  0.0297  [ACCEPTABLE, under 0.10 threshold]

    IMPORTANT: Training F1 is only 3 points above test F1.
    The model generalizes well — it did NOT memorize the training set.
    In v2, this gap was ~7 points; in v3 it is ~3 points.

  Training data: normal + attack (labeled)
  Test data:     normal + attack (labeled)

5.2 Isolation Forest (UNSUPERVISED)
-------------------------------------
  Algorithm: Builds many random trees. Points that are isolated
  (separated from others) quickly → anomaly.
  Works because anomalies are RARE and DIFFERENT from normal data.

  Why unsupervised?
    Trains on NORMAL data ONLY.
    Does NOT need labeled attacks.
    Detects anything unusual compared to learned normal behavior.

  Contamination + max_samples tuning (v3):
    Tries contamination = [0.03, 0.05, 0.08, 0.10, 0.15]
         x max_samples  = ["auto", 0.7, 0.9]  (15 combinations)
    Evaluates each by measuring FPR on a held-out normal validation split.
    Picks the combination where actual FPR is closest to contamination value.
    Selected: contamination=0.10, max_samples=auto

  Test metrics:
    F1=0.8177, Recall=0.9142, FPR=0.1572
    (High recall — catches 91% of attacks — but higher false alarms than RF)

  Training data: normal ONLY
  Test data:     normal + attack (for evaluation)

5.3 Autoencoder (UNSUPERVISED)
--------------------------------
  Algorithm: Neural network that COMPRESSES then RECONSTRUCTS its input.

  How it works:
    1. Train on NORMAL data only
    2. Network learns to compress and reconstruct normal traffic
    3. On new data: compute reconstruction error (MSE)
    4. Normal data → low error (network knows how to reconstruct it)
    5. Attack data → HIGH error (unfamiliar pattern → poor reconstruction)
    6. If error > threshold → ANOMALY

  Architecture comparison (v3 — dropout + L2 regularization):
    Four configs compared by validation loss:
      arch=(32,16) dropout=0.1 l2=1e-4 batch=32  → val_loss=0.0575 (SELECTED)
      arch=(32,16) dropout=0.1 l2=1e-4 batch=64  → val_loss=0.0711
      arch=(32,16) dropout=0.2 l2=1e-4 batch=64  → val_loss=0.1427
      arch=(64,32,16) dropout=0.1 l2=1e-4 batch=64 → val_loss=0.0853
    Selected: (32,16) dropout=0.1 batch=32

  Threshold (v3):
    Three options computed on training reconstruction errors:
      p95:   0.1194 — balanced (SELECTED)
      p97.5: 0.1480
      p99:   0.1807
    Selected: p95 (good balance of recall vs false positive rate)

  Key parameters:
    epochs=80, early_stopping patience=7
    validation held out from normal train (15%)
    dropout=0.1 and L2=1e-4 prevent memorization of noise

  Test metrics:
    F1=0.8283, Recall=0.8283, FPR=0.0839
    ROC-AUC=0.9335

  Training data: normal ONLY
  Test data:     normal + attack (for evaluation)


────────────────────────────────────────────────────────────────
6. EVALUATION  (src/evaluate_models.py)
────────────────────────────────────────────────────────────────

Metrics explained:

  Metric      | What it means in this context
  ------------|------------------------------------------------------
  Accuracy    | % of all predictions correct
              | CAUTION: misleading if data is imbalanced!
              | A model saying "all normal" can get 70% accuracy
              | while missing every attack.
  Precision   | Of all "attack" predictions, how many were real attacks?
              | High precision = few false alarms
  Recall      | Of all real attacks, how many did we catch?
              | High recall = few missed attacks (critical for security!)
  F1-score    | Harmonic mean of precision and recall.
              | Best single metric for imbalanced classification.
  ROC-AUC     | Area under the ROC curve. 0.5 = random, 1.0 = perfect.
  FPR         | False Positive Rate: normal traffic flagged as attack.
              | We want this LOW to avoid blocking legitimate users.

  Best model selection:
    Winner = highest F1-score WITH recall >= 0.60 AND FPR <= 0.30
    Reasoning:
      - Security requires high recall (catch attacks, not miss them)
      - FPR <= 0.30 ensures we don't block too many real users
      - F1 balances both precision and recall
    NOT just accuracy.

6b. THRESHOLD TUNING  (src/threshold_tuning.py)
────────────────────────────────────────────────────────────────

The Random Forest outputs a PROBABILITY (0.0 to 1.0) for each window.
The default threshold of 0.5 says: "if P(attack) >= 0.5 → flag it."

We can adjust this threshold to trade off between recall and false alarms:
  Lower threshold → flag more → higher recall, more false positives
  Higher threshold → flag less → fewer false positives, some missed attacks

Threshold tuning table (v3):
  Threshold  Accuracy  Precision  Recall   F1      FPR     TP   FP   FN
  0.25       0.8817    0.7508     0.9571  0.8415  0.1551  223   74   10
  0.30       0.8958    0.7789     0.9528  0.8571  0.1321  222   63   11
  0.35       0.9028    0.7950     0.9485  0.8650  0.1195  221   57   12
  0.40       0.9155    0.8314     0.9313  0.8785  0.0922  217   44   16
  0.45       0.9282    0.8699     0.9185  0.8935  0.0671  214   32   19
  0.50       0.9408    0.9064     0.9142  0.9103  0.0461  213   22   20  (default)
  0.55       0.9451    0.9217     0.9099  0.9158  0.0377  212   18   21  <-- SELECTED
  0.60       0.9437    0.9289     0.8970  0.9127  0.0335  209   16   24

Selection criterion: recall >= 0.88 AND FPR <= 0.05, maximize F1
Selected threshold: 0.55 (F1=0.9158, Recall=90.99%, FPR=3.77%)

Why 0.55?
  At default 0.5: FPR=4.6%, F1=0.9103
  At tuned 0.55:  FPR=3.8%, F1=0.9158
  We gain both: fewer false alarms AND slightly better F1.

Saved: results/best_threshold.json, results/threshold_tuning.csv
Plot:  results/figures/threshold_tradeoff.png

6c. ERROR ANALYSIS  (src/error_analysis.py)
────────────────────────────────────────────────────────────────

After training, we analyze WHICH attacks are hardest to detect.

Per-attack-type results (at threshold=0.5):
  Attack Type      Windows  TP   FN  Recall  F1
  flood            16       16    0  1.0000  1.0000  (all caught)
  slow_brute       119     115    4  0.9664  0.9829
  credential       54       49    5  0.9074  0.9515
  recon            44       33   11  0.7500  0.8571  (hardest)

Zero False Positives per attack type (precision=1.0 for all).

Why is recon the hardest?
  Recon (reconnaissance scan) visits all routes with normal-looking latency.
  Some recon windows have only 3 requests with success_rate=1.0 — indistinguishable
  from a normal user who happens to explore all endpoints. The feature that helps
  is requests_per_route=1.0 (each route visited exactly once) combined with
  unique_routes=3 or 4, but in a short window this overlaps with normal behavior.

False Positive analysis:
  22 normal windows flagged as attacks.
  Common cause: normal users who had a burst of 401 errors (e.g., wrong password
  once) combined with higher-than-average latency. These edge cases overlap with
  the brute-force attack signature.

────────────────────────────────────────────────────────────────

Loads the best model and scaler.
Accepts new API log window data (same 9 features).
Outputs:

  🟢 NORMAL
  or
  🔴 ANOMALY DETECTED — high request count (85); many 401 Unauthorized (40); ...

Explanation logic:
  - high request_count → possible DDoS
  - many status_401   → credential stuffing / brute force
  - many status_429   → rate limit triggered (flood)
  - high latency      → server under stress
  - single route      → targeted attack pattern

Usage:
  python src/detect_anomaly.py              # demo samples
  python src/detect_anomaly.py --csv my_logs.csv   # from file

9. TRAFFIC SIMULATION SCRIPTS
────────────────────────────────────────────────────────────────

scripts/normal_traffic.py:
  - 20 normal user sessions
  - Valid login → JWT → /secure + /orders
  - Human-like delays (0.5–3 seconds between requests)
  - Expected HTTP responses: mostly 200/201

scripts/attack_traffic.py:
  - Scenario 1: Flood /public with 30 rapid requests
  - Scenario 2: GET /secure with no token (5× → 401)
  - Scenario 3: GET /secure with 5 invalid tokens (→ 403)
  - Scenario 4: Brute-force login with 7 wrong passwords (→ 401)
  - Scenario 5: POST /orders with 6 malformed JSON payloads (→ 400/500)
  - Scenario 6: POST /orders with 8 invalid payloads (→ 400)

WHY V5 IMPROVED OVER V4 (F1 = 0.920 -> 0.972)
------------------------------------------------
  Five per-IP historical features were added:
    - prev_requests_per_route   : request density in this IP's previous window
    - delta_requests_per_route  : change from the previous window
    - rolling2_error_rate       : mean error rate over the 2 prior windows
    - rolling2_unique_routes    : mean unique routes over the 2 prior windows
    - rolling2_route_entropy    : mean route entropy over the 2 prior windows

  The most impactful is rolling2_error_rate (permutation importance = 0.20).
  Attack IPs generate elevated errors consistently across multiple windows;
  normal IPs do not. This pattern is invisible when analyzing one window at
  a time, but becomes clear across two or more consecutive windows.

LEAKAGE AUDIT -- ALL CLEAN
---------------------------
  1. Rolling features use only past windows.
     shift(1) is applied before rolling(2), so the current window's own
     value is never included in its rolling mean. Verified in code at
     build_dataset.py lines 299-318.

  2. Features are computed on the full dataset, then the train/test split
     is applied. This is correct: test windows' rolling features reference
     their own past windows (which fall in the training period), exactly
     as in production deployment.

  3. Scaler fitted on training data only.
     scaler.fit_transform(X_train) -- scaler.transform(X_test).
     Test statistics never influence the scaler. Verified in preprocess.py.

  4. TimeSeriesSplit validates on future windows in every fold.
     No shuffling. Each fold trains on earlier windows and validates on
     later ones, matching production deployment. Verified in train_models.py.

  5. Labels and attack_type are not in the feature set.
     FEATURE_COLS contains 23 behavioral/rate/history features only.
     label, attack_type, window_start, raw counts (request_count,
     status_401_count, etc.) are all excluded. Verified by code inspection
     and runtime column check.

  6. IP addresses are not memorized.
     IP is dropped from the dataset before saving (build_dataset.py:320).
     The model receives no IP string. Rolling features summarize behavior,
     not identity.

MAIN LIMITATION
----------------
  The dataset is synthetic. Each attack IP exclusively attacks, giving its
  rolling features a perfectly consistent signal. In real production logs,
  an attacker might use a machine that also has legitimate prior traffic,
  or a legitimate user might have a burst of failed logins. In those cases,
  rolling2_error_rate would be less discriminative. The methodology is
  sound; the 97.2% F1 reflects a clean synthetic scenario and would likely
  be somewhat lower on real-world data.

ROOT CAUSE (two factors):

     Attack traffic:  8 IPs in a 2-hour burst  = ~15 requests/window/IP

  2. Status code gap:
     Normal: status_429_count is ALWAYS 0 (no rate-limit events)
     Attack: status_429_count is ~20% of requests
     One binary check catches every attack window.

  CONCLUSION: The original dataset did not test the model's ability
  to detect anomalies. It tested whether the model can read a value.
  This would fail completely with real production data.
